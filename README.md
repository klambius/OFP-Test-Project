## Описание проекта

Проект содержит набор ручных и автоматизированных тестов для проверки пользовательского функционала формы регистрации на странице:

[https://demoqa.com/register](https://demoqa.com/register)

Автоматизированные тесты реализованы с использованием:

* Python 3.10+
* Pytest
* Selenium WebDriver
* ChromeDriver
* Архитектурный паттерн: Page Object Pattern

---

## Автоматизированные сценарии

В проекте автоматизированы следующие тест-кейсы:

* TC-01 — Успешная регистрация с валидными данными
* TC-06 — Валидация логина (логин начинается с цифры)
* TC-09 — Минимальная длина пароля

Сценарии реализованы в двух вариантах:

* как классические Pytest UI тесты
* как BDD сценарии (pytest-bdd + Gherkin)

Всего выполняется **6 автотестов**:  
3 обычных UI теста и 3 BDD теста, покрывающих те же самые сценарии.  
Это сделано для демонстрации двух подходов к автоматизации в рамках одного набора проверок.

Выбор сценариев обусловлен их критичностью, стабильностью и высокой ценностью при регрессионном тестировании.

---

## Особенности демонстрационного прогона

Один из тестов (валидация логина, начинающегося с цифры) в текущем прогоне падает намеренно.

Это сделано в рамках тестового задания для демонстрации:

* корректной фиксации падения теста
* отображения ошибки в отчётах
* работы CI и отчётности при наличии дефектов

Таким образом показывается, что система тестирования отслеживает проблемные сценарии, а не только успешные прогоны.

---

## Особенности тестирования CAPTCHA

Механизм защиты от ботов (CAPTCHA) не автоматизируется, так как:

* CAPTCHA предназначена для предотвращения автоматических действий
* автоматизация CAPTCHA противоречит её назначению

В автотестах проверяется невозможность отправки формы без прохождения защиты.


## Запуск тестов

Запуск всех автотестов:

```bash
pytest
```

Запуск с отображением браузера:

```bash
pytest -s
```

## CI

В репозитории настроен CI-прогон автотестов.

При изменениях в проекте автоматически запускается набор тестов, что позволяет:

* быстро выявлять регрессии
* контролировать стабильность автотестов
* проверять воспроизводимость результатов прогона
